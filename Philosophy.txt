# OpenAnim Architecture Notes

OpenAnim is fundamentally not an “AI video generator” in the same category as diffusion or autoregressive video systems like Sora. Architecturally, it is much closer to a deterministic multimodal compilation engine where natural language acts as source input, orchestration acts as compilation, execution providers act as runtimes, and render artifacts become the final compiled output. Thinking about the system this way is extremely important because it changes how the engine should be designed internally. The system should behave more like LLVM, Bazel, Godot, or a build pipeline than a chatbot wrapper around Manim.

The project is split into two separate repositories. The public repository, `openanim/openanim`, is the actual engine and should remain fully open source, local-first, deterministic, and provider-agnostic. Users should be able to clone it, attach their own LLM provider such as OpenAI, Anthropic, Ollama, or vLLM, and generate animations locally without any dependency on OpenAnim Cloud. The private repository, `openanim/openanim-cloud`, is only the hosted SaaS platform that wraps around the OSS engine and adds distributed rendering, queues, authentication, storage, orchestration services, analytics, billing, and hosted APIs. The cloud repo should consume the OSS engine as a versioned package dependency rather than embedding it through git submodules because package-based integration gives cleaner versioning, rollback support, CI/CD isolation, and stable deployment boundaries.

The single most important architectural component in the entire engine is the canonical scene IR. This should be developed before agents, orchestration, or advanced rendering logic. The biggest mistake most systems make is directly generating renderer-specific code like Manim Python scripts as the source of truth. Instead, OpenAnim should generate and maintain a renderer-agnostic semantic intermediate representation containing scenes, nodes, animations, timelines, assets, transitions, metadata, and execution constraints. The IR becomes the permanent editable source of truth while renderers become disposable execution providers. This is essentially the same architectural idea used in LLVM where frontends compile into IR before backends generate machine code. Without a proper IR, the entire system eventually collapses into renderer-coupled prompt spaghetti that becomes impossible to patch incrementally.

Immediately after the IR layer, the next critical component is the scene graph runtime. The engine needs a persistent in-memory representation of projects that tracks node hierarchies, timelines, dependencies, relationships, asset references, and mutations. This scene graph is what eventually enables deterministic incremental rendering because the engine can precisely determine which nodes changed and only invalidate the affected render segments instead of recompiling the entire project. This is one of the largest strategic advantages OpenAnim can have over purely generative systems because edits become semantic patches instead of full video regenerations. The architecture here should borrow heavily from game engines like Godot or Blender where everything is represented as editable node trees and dependency graphs.

Once the scene graph exists, the next important layer is IR versioning and diffing. Before adding sophisticated AI orchestration, the engine should support scene patching, rollback snapshots, immutable revisions, and deterministic diffs between project states. This is necessary because the future editing workflow depends on localized modifications rather than full regeneration. A user saying “make this transition smoother” should ultimately translate into semantic graph mutations instead of rebuilding the entire scene. The system therefore starts behaving much more like a source-controlled compiler runtime than a one-shot generation engine.

Only after the IR and scene graph systems are stable should renderer adapters be introduced. Providers such as Manim, Remotion, Mermaid, PlantUML, and FFmpeg should not be treated as monolithic “backends.” They should instead be viewed as execution providers capable of handling different classes of rendering workloads. One scene may use Manim for mathematical visualization while another scene uses Remotion for cinematic UI transitions and another uses Mermaid for diagrams. The abstraction therefore evolves toward provider execution contracts such as `provider.execute(task)` rather than tightly coupled backend render functions. This separation is extremely important for future multi-provider orchestration.

Before actually executing renderer code, the engine needs a local execution sandbox. Since OpenAnim will execute AI-generated code, it should always assume renderer execution is potentially unsafe. Every provider should eventually run in isolated Docker containers with filesystem isolation, timeout controls, process management, stdout/stderr capture, dependency pinning, and resource limits. The difficult engineering challenge in these systems is usually not orchestration itself but deterministic reproducibility across unstable rendering ecosystems involving FFmpeg, Cairo, Chromium, Node.js, LaTeX, GPU drivers, fonts, and Python dependencies. Provider-specific containers help isolate this instability and create reproducible execution environments.

The first actual renderer adapter should be Manim because it already aligns closely with OpenAnim’s deterministic rendering philosophy. Initially, the goal should not be sophisticated orchestration but simply proving the pipeline: IR → provider compilation → sandboxed rendering → artifact output. Once this works reliably, artifact caching and incremental rendering become the next priority. Every scene, node graph, and render configuration should be content-hashed similarly to build systems like Bazel so that previously rendered outputs can be reused without recomputation. The render engine should track dirty nodes and invalidate only affected dependency branches. This eventually becomes one of the strongest differentiators of the entire platform because it enables extremely fast iterative editing.

Only after the deterministic rendering substrate is stable should the project seriously invest into agent orchestration. Most AI infrastructure projects fail because they begin with “multi-agent” systems before establishing a reliable execution substrate. The agent runtime should therefore arrive relatively late in development. Initially, a single orchestrator agent capable of structured planning, execution coordination, tool invocation, retry handling, and state management is sufficient. Over time this can evolve into more specialized planning agents, repair agents, renderer-selection agents, and semantic editing systems, but these higher-level abstractions should sit on top of stable infrastructure rather than replacing it.

The eventual long-term architecture of OpenAnim starts resembling a hybrid between LLVM, Godot, Bazel, OpenHands, and ComfyUI. LLVM contributes the idea of stable IR and backend abstraction. Godot contributes scene graphs and editable node trees. Bazel contributes deterministic incremental builds and artifact caching. OpenHands contributes autonomous repair loops and execution feedback systems. ComfyUI contributes local-first composable execution graphs and plugin ecosystems. Studying those repositories structurally is extremely valuable because OpenAnim is effectively a synthesis of those architectural ideas adapted for deterministic multimodal animation generation.
